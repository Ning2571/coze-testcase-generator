# 基于Coze的测试用例自动生成工作流

## 项目简介

一个基于 **Coze（扣子）** 零代码平台搭建的测试用例自动生成工具。

**解决的核心问题**：在测试工作中，每次拿到新需求文档后，从零开始整理测试用例大纲是一件重复且耗时的工作，初学者还容易遗漏某个测试维度（如边界值、异常场景）。

**我的方案**：用Coze工作流把“需求文档上传 → AI解析 → 输出测试用例思维导图”这个流程自动化。AI负责快速搭好从0到1的用例框架，测试人员把精力留给复杂的逻辑校验和边界挖掘。

搭建核心流程只用了约半小时，但为了调优系统提示词、让生成的用例贴合真实测试标准，花了约两个小时反复迭代。

## 工作流配置说明

### 工作流全貌

> 4个节点串联：文档上传 → 内容解析 → 大模型生成用例 → 输出思维导图

<img width="1892" height="747" alt="image" src="https://github.com/user-attachments/assets/c7ae24b4-e9d9-47ea-a064-cd8b3bb15ab4" />


### 四个节点的配置要点

| 节点 | 作用 | 关键配置 |
|:---|:---|:---|
| **开始节点** | 接收用户上传的需求文档 | 输入变量类型设为 `document`，支持直接上传Word、PDF格式 |
| **文档读取** | 解析上传文档的全部文本内容 | 使用 `LinkReaderPlugin` 插件，将文档转为可被大模型处理的纯文本 |
| **大模型** | 根据需求内容，按测试标准生成结构化用例 | 模型选 `豆包` 或 `GPT-4o`|
| **格式转换** | 将模型生成的Markdown转为思维导图 | 使用 `get_treemind_by_markdown` 插件，输出可直接打开的思维导图链接 |

## 数据清洗与格式化

生成的用例导入飞书多维表格后，部分行的“测试维度”列出现了格式问题：优先级标记和维度描述混在一起（如 `P1【异常校验】`），无法直接用于筛选和管理。

我没有逐行手动修改，而是直接调用飞书内置AI助理，给出了一条精确的指令：

> “将表格中32至85行、测试维度列里所有包含‘P1【异常校验】’的单元格，删除其中的‘【’、‘】’符号和‘P1’字符，仅保留‘异常校验’。”

几秒内完成全部清洗，表格恢复为标准格式。

对我而言，无论是用Coze搭工作流，还是用飞书AI处理数据，核心逻辑是一样的：**人负责定义清楚标准和指令，AI负责高效执行。**

## 项目价值总结

- **痛点解决**：将“从零写用例大纲”的重复劳动自动化，AI搭框架，人做审核和补充
- **质量约束**：通过精心调试的系统提示词，让AI按“正向/边界/异常”维度和P0/P1/P2优先级产出用例
- **迭代方法论**：先手动建立“基线”版本用例，再用工作流产出与之对比，逐步调优提示词，形成可复用的优化流程
- **工程化落地**：生成的用例导入飞书多维表格，成为可筛选、可跟踪、可回归的测试资产
- **最后防线**：每一条用例都必须经过人工审核，这是基于我两次实习审核经验定下的质量底线
